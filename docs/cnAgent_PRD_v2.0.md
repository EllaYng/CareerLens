## 1. 产品背景与痛点定义

### 1.1 市场背景

2024–2025 年 AI 产业高速扩张，大量求职者涌入 AI 相关岗位赛道，但同期也出现了求职工具同质化严重的问题。市面上主流 AI 求职助手大多停留在"改改简历措辞""泛化打分"的初级阶段，无法真正解决"求职者与岗位之间的精准供需匹配"问题。

### 1.2 核心痛点（三大悬空）

通过对求职者使用现有 AI 求职工具的行为观察，识别出以下三类高频痛点：

**痛点一：岗位定义悬空**

传统工具要求用户输入岗位名称（如"AI产品经理"），但同一岗位名称在不同公司、不同业务阶段的实际要求差异极大。仅凭岗位名称生成的分析结果对所有人都适用，对具体求职者毫无参考价值。

**痛点二：算法结果黑盒化**

系统直接输出"匹配度 72%"，用户无从感知计算依据和维度拆解，极易产生"AI 在随机生成数字"的负面认知，工具可信度大幅折损。

**痛点三：成长路线鸡汤化**

输出建议停留在"建议学习 SQL""了解大模型原理"等模糊层面，缺乏可执行的具体步骤和可量化的验收标准（KPI），用户拿到报告后不知道下一步到底做什么。

### 1.3 问题本质

上述三个痛点的共同根因是：**现有工具以"生成内容"为目标，而不是以"帮用户做出行动决策"为目标**。这导致工具的产出物停留在信息展示层，而非决策支撑层。

## 2. 产品定位与核心价值主张

### 2.1 一句话定位

> **求职情报站是一款供需精准匹配型 AI 职业诊断工具**，通过强制对齐具体 JD 与个人简历，将 AI 评估过程透明化、将成长建议 KPI 化，帮助 0–3 年求职者从"看了感觉有用"升级为"用完能立即行动"。

### 2.2 核心差异化（与竞品对比）

| 维度 | 传统 AI 求职工具 | 求职情报站 v2.0 |
|------|----------------|---------------|
| 输入精度 | 岗位名称（泛化） | 完整 JD 文本 + 简历 PDF（精准） |
| 评分可信度 | 黑盒数字，无依据 | 公示加权公式 + 维度得分明细表 |
| 建议可执行性 | "学习 SQL" | "完成 15 道中等 SQL 题，截图提交" |
| 技术透明度 | 无 | 显示推理逻辑和计算过程 |
| 隐私安全 | 数据上传云端 | 本地部署，会话内存流转，不落盘 |

## 3. 目标用户与使用场景

### 3.1 核心用户画像

**主要用户：转型求职者**
- 年龄：22–28 岁
- 背景：理工科或商科本科 / 研究生，有意转入 AI 相关产品 / 运营 / 数据岗位
- 核心诉求：知道自己与目标岗位的真实差距，以及最优先补的能力是什么
- 使用频率：每更换一个目标岗位 JD 使用一次

**次要用户：校招应届生**
- 背景：即将毕业，同时投递多个方向，需要快速评估优先级
- 核心诉求：找到自己最有胜算的方向，集中火力准备

### 3.2 典型使用场景

**场景 A（高频核心场景）**：求职者在 BOSS 直聘看到一个心仪职位，复制 JD 文本，上传简历 PDF，5 分钟内得到该岗位的能力拆解、个人匹配度报告和两周内的针对性学习计划。

**场景 B**：求职者正在准备两个方向（AI PM vs 数据分析），分别输入两个 JD 进行诊断，对比两份报告的 Gap 清单，决定优先投哪个方向。

**场景 C**：面试准备阶段，求职者将目标公司 JD 输入产品，重点查看"JD 高频关键词"和"典型业务场景"模块，提炼面试前需要重点准备的知识点。

## 4. 产品设计目标与成功指标

### 4.1 设计目标

| 目标 | 说明 |
|------|------|
| 可信度 | 用户看完分析报告后，相信评分依据是可追溯的 |
| 可执行性 | 用户拿到行动计划后，能在 10 分钟内开始第一个具体任务 |
| 稳定性 | 大模型输出异常时系统不崩溃，有优雅降级机制 |
| 数据安全 | 用户简历数据不离开本地运行环境 |

### 4.2 成功指标（北极星指标）

**核心指标**：用户完成一次完整诊断后，点击"重新诊断"或更换 JD 再次使用的比例（复用率）。复用行为代表用户认可了上一次的诊断结果，并主动将产品用于下一个岗位。

**辅助指标**：
- 单次诊断完成率：用户输入完整信息并成功获得报告的比例（目标 > 90%，主要受 API 稳定性影响）
- JSON 解析失败率：因大模型输出异常导致报告渲染失败的比例（目标 < 5%）
- GitHub Star 数量：作为开源项目社区认可度的参考指标

## 6. 功能需求明细

### 6.1 侧边栏：源数据配置中心

**功能描述**：用户输入和文件上传的统一入口，决定后续所有分析的数据质量。

**交互规则**：

| 组件 | 类型 | 是否必填 | 校验规则 |
|------|------|----------|----------|
| 当前资历 | 下拉选择 | 是 | 默认"初级（1-3年）" |
| 目标岗位 JD | 多行文本框 | 是 | 空值提交时 st.error 拦截 |
| 简历 PDF 上传 | 文件上传 | 二选一必填 | 仅支持 .pdf 格式 |
| 简历文本手动输入 | 多行文本框 | 二选一必填 | 与 PDF 上传互为备选 |
| 启动诊断按钮 | Primary Button | — | 双端均为空时禁止触发 API 调用 |

**优先级规则**：若 PDF 上传与文本手动输入同时存在，以 PDF 解析结果为准。

--- 

### 6.2 看板一：岗位画像模块

**核心目标**：帮用户逆向拆解 JD，打破信息差，明晰企业真正的招聘刚需。

**功能点 1：能力大盘雷达图**

- 展现形式：Plotly 交互式五维雷达图，支持鼠标悬停显示具体分值
- 数据来源：AI 根据 JD 内容动态输出 5 个能力维度及其重要性权重分值（满分 100）
- 设计意图：将文字型 JD 转化为可视化能力模型，让用户一眼看出岗位最看重什么

**功能点 2：JD 高频关键词标签**

- 展现形式：卡片标签流（Card Tag）布局，每个关键词独立卡片展示
- 数据来源：AI 从 JD 中提取 5–8 个高频硬性考点关键词
- 设计意图：提炼面试备考重点，用户可直接拿去搜索学习资料

**功能点 3：典型业务场景描述**

- 展现形式：列表卡片，每条描述一个该岗位常见的业务场景
- 数据来源：AI 根据 JD 推断该岗位实际会接触的 2–3 类项目场景
- 设计意图：帮用户在面试前建立对该岗位实际工作的具体想象

--- 

### 6.3 看板二：匹配度深度分析（解密黑盒）

**核心目标**：解决可信度问题，让每一分都有据可查。

**功能点 1：全局匹配度 Banner**

- 展现形式：超大字号（45px）高亮显示最终匹配度百分比
- 视觉设计：紫色系品牌色背景 Banner，左侧 border 强调线
- 数据约束：匹配度 = 各维度加权贡献分之和，需与明细表数学一致

**功能点 2：计算公式公示**

- 展现形式：文字说明，明文展示 AI 生成的加权计算公式
- 示例：`总分 = 产品思维(30%) × 90分 + 数据分析(25%) × 65分 + ...`
- 设计意图：核心差异化设计。公式可见 = 过程透明 = 用户信任建立

**功能点 3：维度得分明细表**

- 展现形式：`st.table` 渲染四列数据表

| 评估维度 | 用户得分 | 权重占比 | 加权贡献分 |
|---------|---------|---------|-----------|
| 产品思维 | 90 | 30% | 27 |
| 数据分析 | 65 | 25% | 16.25 |
| ... | ... | ... | ... |

- **数学强校验**：各项`[加权贡献分] = [用户得分] × [权重占比]`，所有贡献分之和 = 全局匹配度，Prompt 层强制约束此逻辑

**功能点 4：能力 Gap 盘点卡片**

- 展现形式：红色左 border 卡片，每个 Gap 独立展示，附优先级标签（高 / 中 / 低）
- 数量约束：AI 输出 2–3 个核心 Gap，避免信息过载
- 每条 Gap 包含：Gap 名称 + 具体能力欠缺描述 + 优先级

--- 

### 6.4 看板三：周级魔鬼行动路线图

**核心目标**：拒绝鸡汤，每一条建议必须具备可交付性和可验证性。

**功能点 1：周级突击卡片**

- 展现形式：按 Week 1、Week 2 聚合，每周一张卡片
- 每张卡片包含：周主题 + 突击目标 + 核心考点标签 + KPI + 推荐资源

**功能点 2：硬性验收标准（KPI 标签）**【核心刚需】

- 展现形式：绿色高亮独立 KPI 框，视觉上与其他内容明显区分
- **内容约束（Prompt 层强制）**：
  - ✅ 合格示例：`在 LeetCode 完成 15 道中等 SQL 题并截图`
  - ✅ 合格示例：`输出 1 份包含 RAG 框架描述的完整 PRD，字数不低于 1500 字`
  - ❌ 不合格示例：`学习 SQL`（无法验收，拒绝输出）
  - ❌ 不合格示例：`了解大模型`（泛化建议，拒绝输出）

**功能点 3：推荐学习资源**

- 展现形式：蓝色标签流，每个资源独立标签
- 内容：AI 针对该 Gap 推荐的具体课程 / 工具 / 文档链接（非泛化推荐）

## 7. 非功能性需求

### 7.1 大模型输出稳定性（JSON 降级防崩机制）

**风险描述**：大模型偶发以下异常输出，导致 `json.loads()` 报错、前端页面崩溃：
- 输出夹带 Markdown 标记（` ```json ` 包裹）
- 网络波动导致返回残缺字符串
- 大模型"创造性"地在 JSON 外添加前置说明文字

**降级策略（三层防护）**：

| 层级 | 措施 | 实现方式 |
|------|------|----------|
| 第一层 | Prompt 约束 | System Prompt 明确要求"只输出纯 JSON，不含任何 Markdown 标记" |
| 第二层 | 前端清洗 | `.replace('```json', '').replace('```', '').strip()` 强行剥离标记 |
| 第三层 | 异常捕获 | `try-except` 包裹 `json.loads()`，捕获失败后 `st.error` 提示用户重新诊断，程序不崩溃 |

### 7.2 响应时间

- API 推理阶段：预期 5–15 秒（受 DeepSeek API 响应速度影响）
- 前端渲染阶段：< 1 秒
- 用户等待期间：`st.spinner` 显示加载提示，避免用户误判为卡死

### 7.3 兼容性

- 运行环境：本地 Python 3.9+ 环境，支持 macOS / Windows / Linux
- 浏览器：通过 Streamlit 本地 localhost 访问，兼容 Chrome / Safari / Edge
- PDF 兼容：支持标准文字型 PDF，不支持纯图片扫描型 PDF（无文字层）

## 8. 数据与隐私安全策略

### 8.1 数据流转原则

本产品当前版本为**本地部署形态**，采用以下数据安全设计：

| 数据类型 | 处理方式 |
|---------|---------|
| 用户简历 PDF | 仅在会话内存（Session State）中流转，不落盘，不写入数据库 |
| 解析后的简历文本 | 同上，会话结束后自动释放 |
| JD 文本 | 通过 DeepSeek API 传输，受 API 服务商隐私协议约束 |
| API Key | 存储于本地 `.env` 文件，`.gitignore` 强隔离，不上传代码仓库 |

### 8.2 代码仓库安全规范

- `.env` 文件（含 API Key）列入 `.gitignore`，仅提供 `.env.example` 模板
- 不在代码中硬编码任何密钥
- 不在日志或错误信息中输出用户简历内容

## 9. 技术架构概览

### 9.1 技术选型说明

| 组件 | 选型 | 选择理由 |
|------|------|---------|
| 前端框架 | Streamlit | 无需前端工程背景即可快速构建可交互 Web 应用，适合 AI 工具原型 |
| 大模型 API | DeepSeek-Chat | 中文理解能力强，JSON 结构化输出稳定，成本远低于 GPT-4 |
| 数据可视化 | Plotly | 支持交互式雷达图，与 Streamlit 原生集成 |
| PDF 解析 | PyPDF2 | 轻量级，无需本地安装额外依赖，适合处理标准文字型 PDF |
| 环境变量管理 | python-dotenv | 标准化 API Key 管理，防止密钥泄漏 |

### 9.2 Prompt 工程核心设计

**Temperature 策略**：设置为 0.2（接近确定性输出），目的是锁定 JSON 格式稳定性，减少大模型在格式上的"创造性发挥"。

**输出格式约束**：Prompt 中明确指定完整 JSON Schema，包括字段名、数据类型、数组结构，并附上示例值，降低大模型的格式误判率。

**数学一致性约束**：在 Prompt 中显式要求"各项加权贡献分之和必须等于 match_score"，将数学逻辑约束前置到推理层。

## 10. 版本迭代路线图

### 10.1 已发布版本

**v1.0（已完成）**
- 基础功能：输入岗位名 + 简历文本，输出泛化匹配分析
- 局限：岗位定义模糊，评分无依据，建议无 KPI

**v2.0（当前版本，已完成）**
- 核心升级：强制输入完整 JD 文本、评分算法透明化、周级 KPI 行动计划
- 新增：PDF 简历解析、Plotly 交互式雷达图、JSON 降级防崩机制

### 10.2 规划中版本

**v3.0：多轮对话 + 个性化追踪（规划中）**

核心方向：从"一次性诊断工具"升级为"持续陪伴式职业教练"。

| 功能模块 | 描述 | 优先级 |
|---------|------|-------|
| 多轮追问对话 | 用户可针对诊断报告的任意条目发起追问，Agent 保持上下文连贯回答 | 高 |
| 学习进度打卡 | 用户可标记 KPI 完成情况，系统动态更新匹配度进度条 | 高 |
| 多 JD 横向对比 | 同时输入多份 JD，输出"推荐优先投递顺序"和各岗位 Gap 对比热力图 | 中 |
| 简历自动优化建议 | 基于 Gap 分析，直接给出简历对应描述的改写建议 | 中 |
| 面试题预测模块 | 根据 JD 关键词，生成该岗位高概率面试题目及参考答案框架 | 低 |

**v3.0 核心产品判断**：v2.0 解决了"用户不知道自己差什么"的问题，但没有解决"用户知道了也做不到"的执行力问题。v3.0 的打卡和追踪机制，本质上是把工具从"诊断仪"升级为"训练营教练"，通过持续陪伴提升用户的实际行动转化率。

**v4.0：SaaS 化与商业变现探索（远期规划）**

| 功能模块 | 商业逻辑 |
|---------|---------|
| 免费版 | 每月 3 次完整诊断，JD 长度 ≤ 500 字 |
| Pro 版（订阅制）| 无限次诊断 + 多 JD 对比 + 历史诊断记录存储 + 长简历解析（Context Window 解锁） |
| 企业版 | 为猎头 / 培训机构提供批量候选人诊断 API 接口 |

**v4.0 核心产品判断**：Context Window 大小是天然的用户分层工具——免费用户使用轻量级短文本分析，付费用户解锁长文档解析和历史记忆能力。这一分层逻辑不需要人为设置功能阉割，而是利用大模型本身的技术特性进行自然分层，降低用户对"付费墙"的抵触感。

### B. 术语表

| 术语 | 定义 |
|------|------|
| JD | Job Description，职位描述，招聘方发布的岗位需求文本 |
| Gap | 用户当前能力与岗位要求之间的能力差距 |
| KPI | Key Performance Indicator，关键绩效指标，此处指学习计划的可验收交付物 |
| RAG | Retrieval-Augmented Generation，检索增强生成，大模型技术方案之一 |
| Temperature | 大模型推理参数，控制输出随机性，值越低输出越稳定 |
| Session State | Streamlit 的会话状态管理机制，用于在用户交互间保存数据 |
