## 0. v2.1 复盘与修订说明

本版本基于v2.0发布后的复盘和2次用户访谈，对原文档的几个关键判断做了修正。修订不是推翻重写，是保留发现问题、推翻假设、重新验证的完整过程，原因和过程详见各章节标注。

**本次修订的三个主要发现：**

1. **用户痛点的来源需要被诚实地说明。** v2.0中"通过对求职者行为观察识别出三大痛点"这一描述不准确——三个痛点实际来自作者本人使用v1.0时的真实体验，尚未经过其他用户验证。v2.1已修正措辞，并补充了访谈验证计划与初步结果。
2. **"主要用户/次要用户"的身份划分（转型求职者/校招应届生）经不起推敲。** 产品的实际机制（要求输入完整JD）对任何身份背景的用户都一样，真正决定用户能否用好产品的是"是否已锚定具体岗位JD"这一行为状态，不是身份标签。v2.1已重新定义目标用户，并明确了当前产品边界。
3. **北极星指标"复用率"是滞后指标，无法反映核心价值是否真正发生。** v2.1按"厘清认知→明确决策→真正行动"三层价值链条，重新设计了指标体系。

此外，在2次非正式用户访谈中发现了一个此前未被发现的技术bug（大模型计算的加权分与总分数学不一致）；另有两个产品体验问题（JD输入门槛、PDF上传上限配置疏漏）是作者自用产品时发现的，与访谈无关，均已处理，详见第6、7章节。

## 1. 产品背景与痛点定义

### 1.1 市场背景

2024–2025 年 AI 产业高速扩张，大量求职者涌入 AI 相关岗位赛道，但同期也出现了求职工具同质化严重的问题。市面上主流 AI 求职助手大多停留在"改改简历措辞""泛化打分"的初级阶段，无法真正解决"求职者与岗位之间的精准供需匹配"问题。

### 1.2 核心痛点（三大悬空）

> **【v2.1 修订说明】** 以下三个痛点最初的来源是作者本人使用 v1.0 时的真实体验，不是经过系统性用户研究得出的结论。这是当前产品最大的局限，在此明确说明，而非回避。2026年6月已针对此进行了2次非正式用户访谈，初步结果见本节末尾。

**痛点一：岗位定义悬空**

v1.0 要求用户输入岗位名称（如"AI产品经理"），但同一岗位名称在不同公司、不同业务阶段的实际要求差异极大。作者本人在使用过程中发现，仅凭岗位名称生成的分析结果对所有人都适用，对具体求职者毫无参考价值。

**痛点二：算法结果黑盒化**

v1.0 直接输出"匹配度 72%"，但不展示计算依据和维度拆解。作者本人在使用时产生了"AI 在随机生成数字"的怀疑，难以判断该不该信。

**痛点三：成长路线鸡汤化**

v1.0 输出建议停留在"建议学习 SQL""了解大模型原理"等模糊层面，作者本人在尝试执行时，不知道要学到什么程度（入门还是精通）、时间有限时该如何取舍。

**初步访谈结果（2例，非正式验证）**：

2位受访者（均非AI相关岗位求职者）中，1位主动用强烈措辞描述了"匹配度不确定"带来的焦虑（"gap一年跟触犯天条一样"），另1位面对类似不确定性时的应对方式是"直接投，不纠结"——说明痛点强度可能因人而异，不能假设所有目标用户都同等焦虑，这是后续需要更大样本验证的方向。

关于"黑盒化"痛点，访谈有一个重要的修正发现：**维度拆解+公式公示这一设计，并未像预期那样提升两位用户的信任度**。但深入追问后发现，根源不是"过程不透明"，而是更底层的"对AI判断能力本身的怀疑"（"AI可能不太清楚这个岗位的真实情况，这只是前期预测"）。这意味着v2.0对"黑盒化"痛点的理解可能不够准确，真正的信任问题发生在比"过程透明度"更上游的位置，需要重新设计验证方式。详见附录C《用户访谈纪要摘要》。

### 1.3 问题本质

上述三个痛点的共同根因是：**现有工具以"生成内容"为目标，而不是以"帮用户做出行动决策"为目标**。这导致工具的产出物停留在信息展示层，而非决策支撑层。

## 2. 产品定位与核心价值主张

### 2.1 一句话定位

> **CareerLens 是一款供需精准匹配型 AI 职业诊断工具**，通过强制对齐具体 JD 与个人简历，将 AI 评估过程透明化、将成长建议 KPI 化，帮助 0–3 年求职者从"看了感觉有用"升级为"用完能立即行动"。

### 2.2 核心差异化（与竞品对比）

| 维度 | 传统 AI 求职工具 | CareerLens v2.0 |
|------|----------------|---------------|
| 输入精度 | 岗位名称（泛化） | 完整 JD 文本 + 简历 PDF（精准） |
| 评分可信度 | 黑盒数字，无依据 | 公示加权公式 + 维度得分明细表 |
| 建议可执行性 | "学习 SQL" | "完成 15 道中等 SQL 题，截图提交" |
| 技术透明度 | 无 | 显示推理逻辑和计算过程 |
| 隐私安全 | 数据上传云端 | 本地部署，会话内存流转，不落盘 |

## 3. 目标用户与使用场景

### 3.1 核心用户画像

> **【v2.1 修订说明】** v2.0原本按"主要用户：转型求职者 / 次要用户：校招应届生"划分，这个划分套用了常见PRD模板，但经不起推敲：产品的核心机制是"输入完整JD→评分→行动建议"，这个输入门槛对任何身份背景的用户都一样，决定用户能否用好产品的不是"应届还是转型"，而是"是否已经锚定了一个具体岗位的JD"。v2.1改用行为状态重新定义目标用户。

**目标用户：已锚定具体岗位JD、正在准备投递或面试的求职者**

- 核心特征：手里有一份明确的目标岗位JD，但不确定自己与该岗位的真实匹配程度，也不清楚下一步该补什么
- 不区分应届 / 转型背景，因为两者在"拿着JD分析匹配度"这一行为模式上没有本质区别
- 核心诉求：知道自己与目标岗位的真实差距，以及最优先补的能力是什么

**当前产品边界（明确不服务的人群）**：还在探索求职方向、没有具体JD的用户。这是当前版本明确的产品边界，不是疏漏——若要服务这群人，产品逻辑要从"JD匹配诊断"变为"方向探索"，是完全不同的产品形态，会让现有工具变得臃肿。

**该边界的两个触发拓宽条件**：① 核心功能（JD匹配诊断）被验证确实对目标用户有效；② 后续验证中发现现有用户里有相当比例其实并无明确方向，只是勉强找了个JD来试用。在这两个条件出现之前，暂不规划"方向探索"模块。

### 3.2 典型使用场景

**场景 A（高频核心场景）**：求职者看到一个心仪职位，复制 JD 文本，上传简历 PDF，5 分钟内得到该岗位的能力拆解、个人匹配度报告和两周内的针对性学习计划。

**场景 B**：求职者正在准备两个方向（如 AI PM vs 数据分析），分别输入两个 JD 进行诊断，对比两份报告的 Gap 清单，决定优先投哪个方向。

**场景 C（访谈中发现的新场景，原设计未覆盖）**：用户已拿到面试机会，距离面试时间较紧（如仅剩两三天），来不及真正补齐能力差距，于是将目标公司JD输入产品，重点查看Gap分析，提前预判面试官可能追问哪些薄弱环节，想好该如何回应，而不是用计划去"提升"这些短板。这与场景A的本质区别是**使用时机和诊断结果的用途不同**：场景A发生在投递前，诊断结果用于指导"提升行动"（去补能力）；场景C发生在已获得面试机会之后，诊断结果用于"预判与应答准备"（时间上来不及真正补齐，只能想好怎么回应）。此场景由用户在访谈中主动提出，原设计（行动路线图/两周计划）只覆盖了"投递前提升"这一种用途，未覆盖"面试前应答准备"这一用法，值得作为后续方向探索（详见附录C）。

## 4. 产品设计目标与成功指标

### 4.1 设计目标

| 目标 | 说明 |
|------|------|
| 可信度 | 用户看完分析报告后，相信评分依据是可追溯的 |
| 可执行性 | 用户拿到行动计划后，能在 10 分钟内开始第一个具体任务 |
| 稳定性 | 大模型输出异常时系统不崩溃，有优雅降级机制 |
| 数据安全 | 用户简历数据不离开本地运行环境 |

### 4.2 成功指标（北极星指标）

> **【v2.1 修订说明】** v2.0原本用"复用率"作为北极星指标，但这是个滞后指标——它只能反映用户大概觉得产品有点用，无法说明产品的核心价值（厘清认知→明确决策→真正行动）究竟在哪一层真正发生。v2.1按价值链条重新设计了三层指标体系。

**核心价值链条**：用户使用产品的过程分三层发生——① 厘清认知（看清自己和岗位的差距）→ ② 明确决策（知道下一步该做什么）→ ③ 真正行动（去执行）。北极星指标应紧贴这条链条，而不是紧贴"用户会不会再来"这类留存类指标。

| 层级 | 指标 | 说明 |
|------|------|------|
| 厘清层 | 用户对"评分是否认可"的自评星级（1-5星） | 自我报告类指标，可信度有限，**仅作辅助参考，不纳入正式指标体系** |
| 明确层（核心） | 用户在行动计划页面输入"下一步要做什么"的完成率与内容具体度 | 实际产出，不是自我报告。用户写得出来、写得具体，说明"明确"这一层真正发生 |
| 行动层 | 代理指标：用户是否截图/导出行动计划页面 | 产品目前是一次性工具，无法直接观测用户离开后是否真正执行。打卡功能是更直接的方案，但属于新增开发，会引入新的不确定性（万一打卡功能本身没人用，无法判断是用户没执行还是功能难用）。现阶段先用低成本代理指标，验证方向后再决定是否值得投入开发打卡功能 |

**辅助指标**：
- 单次诊断完成率：用户输入完整信息并成功获得报告的比例（目标 > 90%）。失败来源包括JSON解析失败（占5%，详见下一条）、API超时、网络波动、PDF解析失败等，90%这一目标为后三类因素预留了约5%的缓冲空间，该缓冲空间为估计值，尚未经过实际运行数据验证。
- JSON 解析失败率：因大模型输出异常导致报告渲染失败的比例（目标 < 5%，详见第7.1章节关于该指标的判断依据）

## 6. 功能需求明细

### 6.1 侧边栏：源数据配置中心

**功能描述**：用户输入和文件上传的统一入口，决定后续所有分析的数据质量。

**交互规则**：

| 组件 | 类型 | 是否必填 | 校验规则 |
|------|------|----------|----------|
| 当前资历 | 下拉选择 | 是 | 默认"初级（1-3年）" |
| 目标岗位 JD | 多行文本框 | 是 | 空值提交时 st.error 拦截 |
| 简历 PDF 上传 | 文件上传 | 二选一必填 | 仅支持 .pdf 格式 |
| 简历文本手动输入 | 多行文本框 | 二选一必填 | 与 PDF 上传互为备选 |
| 启动诊断按钮 | Primary Button | — | 双端均为空时禁止触发 API 调用 |

**当前实现逻辑**：若同时上传PDF并在文本框中输入内容，代码会用PDF解析结果**完全覆盖**手动输入内容，手动输入的文字会被直接丢弃，不会合并。

> **【v2.1 新增·自用发现】** 在自己使用产品的过程中发现三个细节问题，性质不同，处理方式也不同：
>
> 1. **JD输入门槛（真实问题，需排期解决）**：部分招聘App（如Boss直聘）不支持站内复制文本，用户需先截图再转文字再粘贴，操作繁琐。这不是单纯的体验小问题，而是一个**会影响后续所有用户验证数据质量的前置漏斗卡点**——若不解决，因输入麻烦而放弃使用的用户根本不会出现在任何统计样本里，导致后续对"核心痛点是否成立"的判断产生偏差。建议方向：支持截图上传+OCR文字识别，优先级与"用户分层修正""自动重试机制"同级（不依赖用户验证结果，可立即排期）。
> 2. **PDF上传上限200MB（技术疏漏，非产品问题）**：经查证为 `st.file_uploader()` 组件未修改的系统默认值，不是经过业务考量设定的数字。这不是真实影响用户体验的问题（没有用户会上传接近200MB的简历），已直接调整为合理范围，不纳入优先级讨论。
> 3. **PDF与手动输入的覆盖逻辑可能丢失真实信息，需先实验验证再决定如何处理**：当前实现是"上传PDF后完全覆盖手动输入内容"，这会导致用户在文本框补充的、PDF简历里没写的信息（如未写入简历的项目经历）被直接丢弃。但改成"PDF文本+手动输入文本直接拼接"也有代价：若用户在文本框中重复粘贴了与PDF相同或部分相同的内容，会增加token成本，更重要的是可能让大模型对重复出现的经历分配过高权重，导致匹配度判断产生不真实的偏移。设计对照实验验证影响大小：对照组为仅PDF，实验组一为PDF+完全重复内容，实验组二为PDF+部分重复内容，以"岗位匹配度/Gap分析/行动计划与对照组的相似度"作为衡量指标。若偏差在可接受范围内，直接改为"两者拼接"；若偏差超出可接受范围，则优化文本框引导文案（引导用户填写"PDF之外的补充信息"而非重复粘贴），降低重复输入发生的概率。该问题关乎核心输出质量，但不属于典型用户行为触发的场景，不会导致用户从访谈/统计样本中消失，归入"立刻可做、不依赖验证结果"的优先级层级。

**核心目标**：解决可信度问题，让每一分都有据可查。

**功能点 2：计算公式公示**

- 展现形式：文字说明，明文展示加权计算公式

- 展现形式：`st.table` 渲染四列数据表

| 评估维度 | 用户得分 | 权重占比 | 加权贡献分 |
|---------|---------|---------|-----------|
| 产品思维 | 90 | 30% | 27 |
| 数据分析 | 65 | 25% | 16.25 |
| ... | ... | ... | ... |

- **v2.1 修订·计算逻辑变更**：大模型仅输出每个维度的原始得分（score）和权重（weight），**加权贡献分与总分均由代码计算**（`weighted_score = score × weight ÷ 100`，`match_score = Σweighted_score`），不再要求大模型自行完成加权计算。原因见7.1章节

**功能点 4：能力 Gap 盘点卡片**

- 每条 Gap 包含：Gap 名称 + 具体能力欠缺描述 + 优先级

## 7. 非功能性需求

### 7.1 大模型输出稳定性（JSON 降级防崩机制）

> **【v2.1 修订说明】** v2.0原本依赖Prompt层约束"各项加权贡献分之和必须等于match_score"，并在代码中做数学一致性校验，校验失败则要求用户重新诊断。但2026年6月的用户访谈中，两位用户均触发了"数学不一致"报错——此前作者自测（标准化JD/简历）未出现过这一问题，说明自测样本的覆盖面不足，真实用户输入的多样性更容易触发模型在数学计算上的不稳定。**根本修复**：将加权分与总分的计算从大模型转移到代码端，大模型只需输出每个维度的原始得分和权重，不再要求其完成乘法和加总，从根本上消除此类失败，而不是仅靠校验和重试去补救。

**风险描述**：大模型偶发以下异常输出，导致 `json.loads()` 报错、前端页面崩溃：
- 输出夹带 Markdown 标记（` ```json ` 包裹）
- 网络波动导致返回残缺字符串
- 大模型"创造性"地在 JSON 外添加前置说明文字
- ~~大模型自行计算的加权分之和与总分不一致~~（v2.1已通过转移计算逻辑至代码端根本解决，不再依赖模型自查）

**降级策略（四层防护 + 代码端计算）**：

| 层级 | 措施 | 实现方式 |
|------|------|----------|
| 第一层 | Prompt 约束 | System Prompt 明确要求"只输出纯 JSON，不含任何 Markdown 标记"，且只需给出原始得分与权重，不要求计算加权分和总分 |
| 第二层 | 前端清洗 | `.replace('```json', '').replace('```', '').strip()` 强行剥离标记 |
| 第三层 | 字段校验 | 校验必要字段是否存在，权重之和是否接近100%（允许±2误差） |
| 第四层 | 代码端计算 | 加权贡献分与总分均由 Python 计算得出，不依赖大模型的数学能力，从根本上避免数学不一致问题 |
| 异常兜底 | 异常捕获 | `try-except` 包裹全流程，捕获失败后 `st.error` 提示用户重新诊断，程序不崩溃 |

**关于 JSON 解析失败率 < 5% 这一指标的判断依据**：该容错率基准是"用户可接受的错误率应与错误后果的严重程度成反比"——本产品场景下出错代价仅为用户重新点击、等待10-15秒，不涉及数据丢失或错失机会，因此5%（约20次诊断中1次需重试）是当前合理的容错水位。若要进一步降至1%，可选手段包括：降低Temperature（当前已为0.2，边际收益有限）、简化JSON Schema（会损害"公式公示"等核心功能，不可行）、增加自动重试机制（可行但增加响应时间和API调用成本，需评估预算）。综合判断，当前5%是较优选择，非随意设定，但保留未来按需调整的空间。

### 7.2 响应时间

- API 推理阶段：预期 5–15 秒（受 DeepSeek API 响应速度影响）
- 前端渲染阶段：< 1 秒
- 用户等待期间：`st.spinner` 显示加载提示，避免用户误判为卡死

### 7.3 兼容性

- 运行环境：本地 Python 3.9+ 环境，支持 macOS / Windows / Linux
- 浏览器：通过 Streamlit 本地 localhost 访问，兼容 Chrome / Safari / Edge
- PDF 兼容：支持标准文字型 PDF，不支持纯图片扫描型 PDF（无文字层）

## 9. 技术架构概览

### 9.1 技术选型说明

| 组件 | 选型 | 选择理由 |
|------|------|---------|
| 前端框架 | Streamlit | 无需前端工程背景即可快速构建可交互 Web 应用，适合 AI 工具原型 |
| 大模型 API | DeepSeek-Chat | 中文理解能力强，JSON 结构化输出稳定，成本远低于 GPT-4 |
| 数据可视化 | Plotly | 支持交互式雷达图，与 Streamlit 原生集成 |
| PDF 解析 | PyPDF2 | 轻量级，无需本地安装额外依赖，适合处理标准文字型 PDF |
| 环境变量管理 | python-dotenv | 标准化 API Key 管理，防止密钥泄漏 |

### 9.2 Prompt 工程核心设计

**Temperature 策略**：设置为 0.2（接近确定性输出），目的是锁定 JSON 格式稳定性，减少大模型在格式上的"创造性发挥"。

**输出格式约束**：Prompt 中明确指定完整 JSON Schema，包括字段名、数据类型、数组结构，并附上示例值，降低大模型的格式误判率。

**v2.1 修订·计算职责划分**：v2.0原本在Prompt中要求"各项加权贡献分之和必须等于match_score"，依赖大模型自行完成数学计算并自查。但大模型在精确数学计算上存在已知短板，访谈中也确实暴露了这一问题。v2.1调整为：**大模型只负责给出每个维度的原始得分和权重（更适合大模型的语义判断类任务），加权计算和总分汇总全部交给代码完成**（更适合代码做的精确计算类任务）。这是一次关于"哪些任务该交给大模型、哪些该交给代码"的技术取舍，原则是：凡是涉及"数字必须精确对得上"的环节，都不交给模型生成，由代码兜底。

### 10.1 已发布版本

### 10.2 规划中版本（v3.0，按依赖关系排序）

> **【v2.1 修订说明】** v2.0原规划直接列出了一批v3.0功能并标注"高/中/低"优先级，但这种优先级判断方式存在问题——"重要性排序"和"依赖关系排序"是两件不同的事。v2.1改用依赖关系重新组织：**不是哪个更重要，而是哪个的成立要看哪个先被验证**。

**第一组：立刻可做，不依赖任何验证结果**

| 事项 | 来源 | 说明 |
|------|------|------|
| 用户分层修正 | 复盘 | 已在本版本完成，详见3.1章节 |
| JD输入门槛优化（支持截图/OCR） | 自用发现 | 不依赖验证结果即可判断必要性——这是会污染后续抽样质量的前置漏斗卡点，越早解决，后续验证数据才越可信 |
| 自动重试机制 | 复盘（技术取舍） | 纯技术优化，与产品方向是否正确无关，可随时排期 |
| PDF上传上限修正 | 自用发现 | 技术疏漏，顺手修复，不需纳入排期讨论 |
| PDF与手动输入的合并逻辑优化 | 自用发现 | 需先做对照实验验证影响大小，再决定改代码逻辑还是改引导文案；重要但不构成依赖关系，不必抢在外部用户验证之前，详见6.1章节 |

**第二组：最优先、决定后续一切方向**

| 事项 | 说明 |
|------|------|
| 外部用户验证（扩大样本） | 当前仅完成2例非正式访谈，样本量远不足以支撑任何结论性判断。后续所有功能投入的合理性，都建立在这一步的结果之上 |

**第三组：验证通过之后才有意义去做**

| 事项 | 依赖关系 |
|------|------|
| 行动层观测（用户是否真正执行计划） | 前提是"用户愿意持续使用产品"——若用户验证显示用户根本不会回来，精确测量执行率没有意义 |
| 黑盒化痛点的真正解法重新设计 | 访谈显示"维度拆解+公式公示"未达到预期效果，需要先理解信任问题的真正根源（详见1.2章节），再决定怎么重新设计，这本身需要更多验证 |
| "行动计划具体程度"的颗粒度差异化 | 访谈发现不同用户对"具体程度"的舒适区不同，是否需要分用户类型调整，需要更大样本验证后才能判断 |

**第四组：验证后可能都不需要做，要看具体数据再决定**

| 事项 | 触发条件 |
|------|------|
| 拓宽产品边界，服务"方向探索型"用户 | 触发条件二选一：① 核心功能被验证确实有效；② 验证发现现有用户中有相当比例其实并无明确方向。两者皆不满足，则此事可能永远不需要做 |
| "面试弱点预警"独立功能方向 | 访谈中用户自发提出的意外使用场景，需要先确认这是个例还是普遍需求 |
| 产品跨行业适用性的正式验证 | 当前仅有2例非AI岗位的初步观察，且无对照实验，需要设计同简历跨赛道对照测试后才能下结论（详见附录C） |

**v3.0 核心产品判断**：v2.0/v2.1的复盘暴露出一个根本问题——产品目前所有的"功能合理性"判断，都建立在样本量极小（甚至为0）的基础上。v3.0最高优先级不是新增功能，而是把"验证"本身当作核心交付物，用更扎实的数据决定后续投入方向，避免在错误的假设上持续构建功能。

| 功能模块 | 商业逻辑 |
|---------|---------|
| 免费版 | 每月 3 次完整诊断，JD 长度 ≤ 500 字 |
| Pro 版（订阅制）| 无限次诊断 + 多 JD 对比 + 历史诊断记录存储 + 长简历解析（Context Window 解锁） |
| 企业版 | 为猎头 / 培训机构提供批量候选人诊断 API 接口 |

**v4.0 核心产品判断**：Context Window 大小是天然的用户分层工具——免费用户使用轻量级短文本分析，付费用户解锁长文档解析和历史记忆能力。这一分层逻辑不需要人为设置功能阉割，而是利用大模型本身的技术特性进行自然分层，降低用户对"付费墙"的抵触感。该判断暂不受本次复盘影响，仍依赖v3.0验证结果决定是否值得投入。

### C. 用户访谈纪要摘要

**说明**：2026年6月完成2例非正式用户访谈，样本量小，本节结论均为初步信号，非定论。完整访谈记录见独立文档《用户访谈纪要》。

**受访者**：2位，均为非AI相关岗位求职者（应用经济专业研一在读、工商管理专业本科应届），与v2.0原定位（AI/产品相关岗位求职者）不完全匹配，但提供了产品跨行业适用性的初步观察机会。

**核心发现**：

1. **痛点强度因人而异**：一位受访者用强烈措辞描述了匹配度不确定带来的焦虑，另一位面对同类不确定性时选择"直接投，不纠结"，说明原PRD假设的痛点不是均匀分布的，需要更大样本判断是普遍现象还是个体差异。

2. **"黑盒化"痛点的解法未达预期，且根因被重新定位**：维度拆解+公式公示设计后，两位用户信任度均无变化；但根因不是"过程不透明"，而是对"AI是否有资格做此类判断"的更底层怀疑。原v2.0对该痛点的理解可能不够准确。

3. **"具体程度"存在用户自定的舒适区**：最初判断一位用户"还需查资料"等同于"计划不够具体"，但深入追问后该用户主动澄清：她不需要逐字逐句的执行步骤，那样反而像被当小孩教；她需要的是清楚的差距诊断，剩下的规划她自己有能力完成。这是一次访谈过程中的自我纠错，提示"具体"不是越细越好。

4. **意外发现——面试弱点预警场景**：一位用户将产品用于"面试前重点准备薄弱环节"，这是原设计未覆盖的使用场景，已纳入3.2章节场景C。

5. **意外发现——技术bug由真实用户触发**：此前自测未出现的"大模型数学计算不一致"问题，在2次访谈中均被触发，说明自测样本覆盖面不足，已在7.1章节完成根本修复。

6. **自用发现——JD输入门槛与PDF上限**：作者自己使用产品过程中发现的两个产品体验细节（与本次访谈无关），性质不同（一个是会污染验证数据的真实卡点，一个是无实际影响的技术默认值疏漏），已分别处理，详见6.1章节。

**待验证事项**：黑盒化痛点的真正解法、行动计划具体程度的用户类型差异、产品跨行业适用性的对照实验、面试弱点预警场景的需求普遍性。均已纳入10.2章节v3.0规划的"第二/三/四组"事项中。
